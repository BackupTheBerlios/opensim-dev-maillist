Yeah really there isn't that much different between the xml and xml2 formats, when I added xml2, it wasn't actually meant so much as a replacement for the old format, just a alternative, but having two formats is confusing, that together with my bad choice of name for the functions, lead to the belief that it the old version shouldn't be used anymore. 

The old version certainly has those extra features on the console command. So if we are going to tell people to use the xml2 functions, then we should copy support for those extra parameters across.
 
Stefan Andersson <stefan at tribalmedia.se> wrote:    .hmmessage P { margin:0px; padding:0px } body.hmmessage { FONT-SIZE: 10pt; FONT-FAMILY:Tahoma }  Basically, the xml2 format is just a rearrangement of some nodes, to make loading/saving a bit easier.
  
 We have been discussing a xml3 format in throes and fits, but never got anywhere.
  
 One of the properties of xml3 would be (optionally) not to include id's at all.
  
 Then such a load/save file would be truly portable.

 In Tribal Server, we actually support loading content off the web into regions, and we have to re-set all the guids in all the objects on each fetch, because they are fetched as new objects into new contexts, and owner et c has to be re-set on all nodes.

 That is what really should be the base scenario for loading/fetching, ie, no id data at all.
