I think that discussion has to be separated; I don't know the reason for those members to be kept private (other than that the plugin 'wasn't intended to be subclassed' which is fair enough) but I can say that generally the only reason to hesitate over going from 'private' to 'protected' is if code is written so that state integrity is pivotal for method execution - something that usually points to the code being brittle in the first place. (Compare with calling virtual members from constructors for a similar concern)

 

I am perfectly aware that this is touching on religion so I won't push the issue further; suffice to say that "in most cases" we should not have a problem with it.

 

For the interested, I suggest you google for religious flame wars on the proper use of "internal" and "sealed" as well. ;-)
