Well I would certainly agree that a blocking queue should only be used 
where a blocking queue is appropriate. But given that proviso, then when 
it is used, the behavior we're talking about here is the correct one. If 
you want to be able to peek, the GetQueueArray method attempts to satsify 
that need. If you cactually want to take an element, if one is there, and 
return otherwise, then a non-blocking dequeue could be provided, but that 
would beg the question why it should be necessary to defeat the monitor's 
queuing mechanism in this way and it's use should be very carefully 
administered.

We could add a boolean to dequeue indicating whether or not the thread is 
willing to wait; but choosing which call to make has to remain the 
responsibility of the programmer. Choosing the wrong one is an error of 
design, not a fault in the notion of blocking queues.

If you want REALLY hard problems to debug, then incorporating mechanisms 
intended to accomodate programmer's mistakes is definitely the way to go.

The BlockingQueue uses Monitor.Pulse which only releases ONE thread for 
execution. The others remain suspended. PulseAll is needed if you want to 
awaken all the waiting threads. So the race condition that you describe 
should not happen (if Mono corrcetly implements Monitor anway). 

The queue AND the lock are private to the BlockingQueue class, so the only 
unsynchronized mechanism for  examining the queue is GetQueueArray which 
returns a copy of the content. In principle It isn't possible for anyone 
to damage the queue. The lock cannot be obtained by anyone not using the 
proper BlockingQueue accessors.
